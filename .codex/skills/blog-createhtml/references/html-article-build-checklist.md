# Future CX HTML Article Build Checklist

Use this checklist when turning Future CX blog copy into a static `content/*.html` page.

## Inputs To Establish

- Article title and display H1.
- URL slug and filename.
- Meta description.
- Category and optional secondary badge.
- Hero or article image path, plus responsive variants if the selected template uses them.
- Published and modified dates. Do not guess historical dates.
- Article section and keywords for JSON-LD.
- Related articles and internal links.
- Whether the article should be added to `content.html`.

## Template Selection

Prefer cloning the nearest current article pattern:

- Use `content/blog-single.html` as template for blogs with hero image, metadata, body content, and related cards.
- Use `content/ecommerce-trends-radar.html` only for interactive or radar-style pages.
- Avoid old templates such as `content/blog-single.old.html` unless explicitly needed for comparison.

## Required Metadata Alignment

Keep these fields consistent:

- `<title>`
- `og:title`
- canonical URL
- `og:url`
- meta description and `og:description`
- `twitter:description`
- `keywords`
- `article:author`
- `article:published_time`
- `article:modified_time`
- `og:type`
- `og:image`
- `twitter:image`
- JSON-LD `mainEntityOfPage`, `headline`, `description`, `image`, `author`, `publisher`, `datePublished`, `dateModified`, `url`, `articleSection`, `keywords`, and `inLanguage`
- Breadcrumb active label
- Visible H1 and intro text

## Page Shell

Preserve:

- `<!doctype html>` and `<html lang="en" data-bs-theme="light">`
- favicon links and theme color metadata
- early theme script that applies dark/light mode
- stylesheet link to `/assets/css/theme.min.css`
- page loading styles
- Google Analytics script
- shared navbar structure and theme switch
- shared footer structure
- back-to-top button and theme/vendor scripts when present in the template

## Body Content

- Convert article copy into semantic headings and paragraphs.
- Keep H2/H3 hierarchy logical. Do not skip levels for visual effect.
- Use Bootstrap/Silicon classes already present in the chosen template.
- Use cards, callouts, tables, or diagrams only when they make the article easier to understand.
- Keep article sections practical: definitions, principles, decision criteria, tradeoffs, implementation steps, risks, and next actions.
- Add source links near claims when external evidence is used.

## Images

- Use `/assets/img/blog/single/...` for article images.
- Confirm referenced image files exist before treating the page as publish-ready.
- For preload images, make sure paths match the actual hero image variants.
- Keep `og:image` and `twitter:image` absolute under `https://www.futurecx.nl`.
- Use meaningful `alt` text for real `<img>` elements.
- Background-image cards need accessible link labels.

## Content Overview Card

When adding the page to `content.html`:

- Reuse the existing `<!-- ITEM -->` article card pattern.
- Use the same article URL, image path, badges, title, and short summary.
- Keep summary concise and aligned with the page meta description.
- Place the new item according to the current ordering convention, usually newest or most relevant first.
- Preserve existing 2-space indentation and local line wrapping.

## Sitemap

- Check whether `sitemap.xml` is manually maintained or generated by build tasks before editing.
- If editing manually, add the canonical URL and a valid `lastmod`.
- Do not invent `lastmod`; use the known publication or modification date.

## Validation

Run when feasible:

```bash
npm run validate
```

Run `npm run build` when the change affects generated assets, sitemap generation, build scripts, or shared site structure.

Before final response, check:

- Links are valid and root-relative where internal.
- External links include `target="_blank"` and `rel="noopener noreferrer"`.
- JSON-LD is valid JSON.
- Metadata dates are consistent.
- New files are referenced from `content.html` if intended.
- Missing assets or unverified claims are reported clearly.
