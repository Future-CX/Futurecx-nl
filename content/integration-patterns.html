Integration Patterns: Some best practices

Enterprise Architecture
October 23, 2025
Updated: November 7, 2025
Author
Martijn van Deel

Integration Patterns are proven, reusable solutions to common challenges that arise when connecting different systems, applications, and data sources.

They describe how systems should interact, exchange data, and handle events in a consistent, reliable way.

For example:

Problem: You need multiple systems to get real-time updates when a change happens.

Pattern: Use a Publish/Subscribe pattern — one system publishes events, and multiple subscribers get notified without direct dependencies.

The Integration Patterns are derived from Integration Principles. Based on the Integration 'scope' a design direction is given.

Categorizing Integrations

The North Star Architecture consists of layers; Channels, Engagement, Integration and Foundation. We will categorize the integrations based on these layers, as patterns will differ per category.

  1. An Enterprise Solution to deliver 'data' to an Engagement Service via an Integration Component.
  2. An Engagement Service to deliver 'data' to an Enterprise Solution via an Integration Component.
  3. A customer facing Channel connected to an Engagement Service.
  4. An Engagement Service that connects to another Engagement Service.
  5. A Channel that directly communicates to the Enterprise Solution via an Integration Component.

Patterns per category

Below, you will find an overview of the integration patterns categorized by their respective layers. Further down, each category is explained in detail with specific examples and guidelines.

FOUNDATION TO ENGAGEMENT

An Enterprise Solution to deliver 'data' to an Engagement Service via an Integration Component.

In pattern 1A the Enterprise solution sends data in a file (.csv) to Seeburger's internal file system. Seeburger reads this file, transforms the data into the API format the Engagement Solution needs and then calls the Engagement Solution's API to send the data.

In pattern 1B the Enterprise solution sends data in a file (.csv) to Seeburger's internal file system. Seeburger reads this file, transforms and filters the data and sends this to the Azure Service Bus. Any Engagement Solution that is 'subscribed' to the topic, receives a notification that new data is available. The Engagement Solution reads the data, transforms it and stores it to the Engagement Solution.

In pattern 1C the Enterprise solution sends data in a JSON format to the Azure Service Bus directly. Any Engagement Solution that is 'subscribed' to the topic, receives a notification that new data is available. The Engagement Solution reads the data, transforms it and stores it to the Engagement Solution.

In pattern 1D the Enterprise solution sends a notification that new data is available to the Azure Service Bus directly. Any Engagement Solution that is 'subscribed' to the topic, receives a notification that new data is available. The Engagement Solution reads the ID from the notification and calls an API to get the full data based on the ID receives. The Engagement Solution receives the full data from the API, transforms it and stores it to the Engagement Solution.

Rule of Thumb

  • Choose 1a when the enterprise source can only export files, you need strong control/validation/transformation, and there’s a single (or few) known targets with stable APIs.
  • Choose 1b when the source is file-only, but you need fan-out to multiple consumers and looser coupling than 1A.
  • Choose 1c when the enterprise system can publish JSON messages/events directly and you want near real-time, pub/sub distribution.
  • Choose 1d when payloads are large/sensitive or you must always fetch the freshest data from the source; consumers can call back via API.

ENGAGEMENT TO FOUNDATION

An Engagement Service to deliver 'data' to an Enterprise Solution via an Integration Component.

In pattern 2A the Engagement Service send the full data to the Azure Service Bus. Seeburger is 'subscribed' to the topic, receives a notification with full data payload. Seeburger reads the data, transforms the data into the API format the Enterprise Solution needs and then calls the Enterprise Solution's API to send the data.

In pattern 2B the Engagement Service sends a notification that new data is available to the Azure Service Bus directly. Seeburger is 'subscribed' to the topic, receives a notification that new data is available. The Engagement Solution reads the ID from the notification and calls an API to get the full data based on the ID received. The Engagement Solution receives the full data from the API, transforms it and stores it to the Engagement Solution.

In pattern 2C the Engagement Service sends send the full data to the Azure Service Bus. A custom Azure Function is 'subscribed' to the topic, receives a notification with full data payload. The custom Azure Function reads the data, transforms the data into the API format the Enterprise Solution needs and then calls the Enterprise Solution's API to send the data.

In pattern 2D the Engagement Service transforms the data in the format the Enterprise Solution API needs. The Engagement Service calls the Enterprise Solution API, exposed via Azure API Manager, to send the data.

Rule of Thumb

  • Choose 2a when you need Seeburger’s governed transformations and auditing, and pushing the entire record is acceptable.
  • Choose 2b when you want small messages on the bus and must fetch the freshest state from the Engagement API at transform time.
  • Choose 2c when you need low latency or bespoke logic that doesn’t fit Seeburger, and you want cloud-native scalability/observability.
  • Choose 2d when a synchronous write is required (e.g., immediate confirmation) and the Engagement service can perform target-format mapping.

CHANNEL TO ENGAGEMENT

A customer facing Channel connected to an Engagement Service.

In pattern 3A the channel, for example a custom headless frontend, connects to the Engagement Service APIs directly.

In pattern 3B the channel connects to a Custom Service, for example an Azure Container App or Azure Function, via the Azure API Gateway.

Rule of Thumb

  • Choose 3a when the channel can consume the Engagement APIs as-is (data shape, auth, rate limits) and no orchestration or cross-system logic is needed.
  • Choose 3b when you need channel-specific shaping, aggregation, or protocol adaptation; you want to hide backend complexity and manage secrets centrally.

ENGAGEMENT TO ENGAGEMENT

An Engagement Service that connects to another Engagement Service.

In pattern 4A the Engagement Service sends data to another Engagement Service via an out-of-the-box standard connector.

In pattern 4B the Engagement Service sends data to a custom Azure Function. The Azure Function transforms the data and sends it to another Engagement Service API.

In pattern 4C the Engagement Service sends send the full data to the Azure Service Bus. A custom Azure Function is 'subscribed' to the topic, receives a notification with full data payload. The custom Azure Function reads the data, transforms the data into the API format the Engagement Service needs and then calls the Engagement Service API to send the data.

Optionally the Service Bus contains only a notification with an ID and the Azure Function calls the Engagement Service API to get the full data based on the ID received.

Rule of Thumb

  • Choose 4a when a vendor-supported out-of-the-box connector exists that meets functional and non-functional needs (throughput, mapping, auth).
  • Choose 4b when you need custom mapping/logic that the standard connector can’t handle, or you must aggregate/sanitize data before calling the target API.
  • Choose 4c when multiple consumers may need the event, you want fan-out and decoupling, or you need near real-time with buffering/retry semantics.

CHANNEL TO FOUNDATION

A Channel that communicates directly with the Enterprise Solution through an Integration Component.

In Pattern 5A, the channel (webshop) connects via a Backend-for-Frontend (BFF) to the Azure API Manager to retrieve data from the Enterprise Solution APIs. These APIs are exposed through Azure API Management, which manages authentication, throttling, and security. The Enterprise Solution APIs return structured data that the channel can directly consume and present.

In Pattern 5B, the channel (webshop) connects via a BFF and an additional Custom Service to the Azure API Manager to request data from the Enterprise Solution APIs. As in 5A, Azure API Management handles authentication, throttling, and security. The Custom Service processes and interprets the API responses before sending them to the channel. This approach is particularly useful when the channel cannot interpret the API response directly or when multiple API calls must be orchestrated to deliver a single, aggregated response to the channel.

Both integration patterns are synchronous and follow a request/response communication model.

Rule of Thumb

  • Choose 5a when the channel can consume the Enterprise API’s data model as-is, and no cross-system orchestration or heavy transformation is needed.
  • Choose 5b when the channel needs aggregated responses, protocol adaptation, or interpretation of Enterprise API outputs; multiple calls must be orchestrated into one payload.

Integration Principles

The Integration Patterns are derived from Integration Principles. Based on the Integration 'scope' a design direction is given.

Integration Principles: Some best practices

Integration Principles are a set of guidelines or best practices that help us design, build, and manage how different systems, applications, and data sources connect and communicate with each other. These principles ensure that integrations are efficient, secure, scalable, and maintainable.

Integration Principles vs Patterns

Integration Principles Integration Patterns
They are high-level guidelines or rules that define how integrations should be approached (e.g., “keep integrations loosely coupled,” “ensure security by design”). They are practical, reusable design templates that describe how to solve common integration problems (e.g., “Message Broker Pattern,” “Publish/Subscribe Pattern”).
Principles are why and what you should aim for — the foundation for making smart design decisions. Patterns are how you implement those principles in real-life scenarios.
Example: "Integrations should be scalable and loosely coupled." Example: "You might use the Queue-Based Load Leveling Pattern to achieve scalability and loose coupling."
Principles help you decide which patterns to use. Patterns help you apply the principles in a consistent, proven way.
