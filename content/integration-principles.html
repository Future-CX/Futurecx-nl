Integration Principles & Best Practices for Modern Architecture

Integration Principles are a set of guidelines or best practices that help us design, build, and manage how different systems, applications, and data sources connect and communicate with each other. These principles ensure that integrations are efficient, secure, scalable, and maintainable.

Integration Principles are our “rules of the road” for connecting systems, making sure they're connected in a way that's reliable, secure, and built to last.

It serves as a starting point for conversation and integration design. As such it should and will be validated when we compare integration options on a case-by-case basis.

From Integration Principles we can create Integration Patterns, which are 'templates' for detail design an integration.

Categorizing Integrations

The North Star Architecture consists of layers; Channels, Engagement, Integration and Foundation. We will categorize the integrations based on these layers, as principles will differ per category.

  1. An Enterprise Solution to deliver 'data' to an Engagement Service via an Integration Component.
  2. An Engagement Service to deliver 'data' to an Enterprise Solution via an Integration Component.
  3. A customer facing Channel connected to an Engagement Service.
  4. An Engagement Service that connects to another Engagement Service.
  5. A Channel that directly communicates to the Enterprise Solution via an Integration Component.

GENERAL

Design Integrations to Scale - Handle Increasing Loads.

Rationale: Integrations should be designed to scale efficiently as business needs evolve, traffic increases, and data volumes grow. If not, they risk becoming bottlenecks—leading to performance issues, system failures, or unnecessary operational costs.

Implications:

  • Favor event-driven, asynchronous, and decoupled patterns such as queues or pub/sub to prevent bottlenecks from synchronous dependencies.
  • Ensure resilience under heavy load by implementing retries with backoff, idempotency, and circuit breakers to avoid cascading failures.
  • Optimize payloads and data access using delta updates, pagination, batching, and caching to reduce processing overhead and redundant calls.
  • Establish observability and auto-scaling by defining key metrics, SLI* and SLO**, implementing tracing, and enabling automated scaling policies.
  • Anticipate consumer growth by managing API versioning, schema evolution, and rate limiting to support an expanding ecosystem without performance degradation.

* Service level indicators: For example HTTP response status, Response time, etc.
** Service Level Objectives: For example 99% successful under 500ms

GENERAL

Minimize system dependencies to enable independent updates and reduce the risk of bottlenecks.

Rationale: We aim to evolve services and APIs while remaining mindful of the consumers they impact. This includes managing changes to API and data contracts, as well as maintenance schedules.

To support this, we must reduce dependencies between systems, enabling independent updates and preventing bottlenecks.

Implications:

  • Define clear ownership by considering responsibility, planning, and workload when determining who manages each integration.
  • Adopt a loosely coupled design so individual components can be maintained and updated without impacting the overall service. Use APIs, events, and contracts instead of direct dependencies between systems.
  • Support independent release cycles so teams can deploy and update their services without waiting for others.
  • Increase resilience by designing fault-tolerant integrations with retries and fallbacks, ensuring failures in one system have minimal downstream impact.
  • Simplify maintenance so that upgrades, patches, and migrations become easier and less disruptive.
  • Improve scalability by allowing each system to scale independently, preventing load or performance issues from spreading across the landscape.

GENERAL

Design integrations for reuse across multiple use cases.

Rationale: Designing integrations for reuse across multiple use cases reduces redundancy, accelerates development, and improves maintainability. Rather than building bespoke integrations for each scenario, a reuse-driven approach creates a more modular, scalable, and cost-effective integration architecture.

Implications:

  • APIs must be designed with versioning as a standard practice to ensure stability and backward compatibility.
  • Follow integration preferences by prioritizing API-based designs using JSON data formats and leveraging out-of-the-box connectors before creating custom ones.
  • Avoid redundancy by preventing bespoke, point-to-point integrations for each scenario. Reuse existing APIs wherever possible, supported by an enterprise-wide integration catalog.
  • Accelerate delivery by reusing proven integrations, reducing development effort and shortening time-to-market for new capabilities.
  • Improve maintainability by minimizing the number of integration variants, which lowers complexity and support overhead.
  • Promote modularity through well-defined contracts and interfaces that abstract system complexities, making integrations easier to extend, adapt, and repurpose.
  • Lower overall cost by reusing shared integration components, reducing both development and operational expenses.

FOUNDATION ENGAGEMENT

Prefer event-driven integrations over point-to-point or polling approaches.

Produce events for meaningful business state changes; let consumers subscribe and react independently.

Rationale: Event-driven designs cut unnecessary polling, decouple producers from consumers, and enable reuse, scalability, and real-time responsiveness. They also create a durable audit trail and make it easier to add new consumers without changing the producer.

Implications:

  • Source systems must be able to emit events or notifications for relevant business state changes.
  • Events should be lightweight, containing identifiers and minimal context; consumers retrieve full details as needed.
  • Systems can react to changes in real time without waiting for batch updates.
  • Multiple consumers can subscribe to the same event stream without changes to the producer.
  • Producers and consumers scale independently, improving flexibility and performance.
  • Loose coupling between systems increases resilience and reduces the impact of downstream failures.
  • The architecture must tolerate eventual consistency, with clear SLAs for data freshness.
  • Event contracts and versioning must be governed to maintain compatibility and data quality.
  • Integrations must include observability features such as tracing, correlation IDs, retries, and dead-letter handling.
  • Security and compliance controls must be applied to event data, including encryption, retention, and access management.

FOUNDATION TO ENGAGEMENT

Prefer existing integrations over building new ones.

Rationale: We avoid reinventing the wheel by first considering existing integrations. If no suitable option is available—or if current solutions are insufficient—we design a new integration.

Implications:

  • Reduce redundancy by avoiding duplication of functionality that already exists within the integration landscape.
  • Accelerate delivery by leveraging proven, existing integrations to shorten time-to-market for new capabilities.
  • Lower cost by minimizing development and ongoing maintenance efforts through reuse.
  • Increase stability by reusing integrations that are already tested and reliable, reducing the introduction of new risks.
  • Promote governance and standardization by ensuring consistent integration patterns and fewer variations across the enterprise.
  • Evaluate existing integrations against the defined design principles to confirm their suitability before reuse.
  • Acknowledge that exceptions will apply when existing integrations are not reusable or fit for purpose. In such cases, create new integrations designed explicitly for future reusability.

ENGAGEMENT

Prefer available connectors first, then accelerators, and only as a last resort use cloud-native custom integrations.

Rationale: We avoid reinventing the wheel by first considering the out-of-the-box integrations offered by solution providers. If these are unavailable or insufficient, we then assess what an accelerator provides before resorting to a bespoke integration built with Azure services.

Implications:

  • Ownership of integrations depends on the chosen pattern, with responsibility lying either with the source system (push) or the target system (pull).
  • Maximize reuse by starting with pre-built connectors, reducing time, cost, and complexity in delivering integrations.
  • Accelerate delivery by using accelerators when standard connectors are unavailable, speeding up development compared to custom builds.
  • Lower maintenance overhead by prioritizing prebuilt and accelerator-based integrations, which are typically easier to support and update than custom cloud-native solutions.
  • Promote governance and cost efficiency by avoiding unnecessary custom integrations, keeping the architecture simpler, more standardized, and predictable.
  • When custom development is required, integrations must be built using Azure services and designed with scalability, observability, and reusability in mind.

ENGAGEMENT

Prefer direct integrations with SaaS platforms.

Rationale: SaaS platforms come with their own complete infrastructures, including components such as CDNs and API gateways. Adding our own API gateway between channels and these platforms is unnecessary and would only introduce additional complexity and performance overhead.

Implications:

  • Reduce complexity by avoiding unnecessary middleware layers that introduce latency and additional failure points.
  • Improve performance through direct API calls that minimize overhead and deliver faster response times.
  • Lower cost by reducing the number of components involved, which decreases infrastructure and maintenance efforts.
  • Ensure alignment with SaaS best practices by leveraging the platform’s native APIs, gateways, and content delivery networks (CDNs).
  • Simplify governance by managing contracts, security, and monitoring directly with the SaaS provider rather than through multiple integration layers.

CHANNEL TO ENGAGEMENT

Use an API Gateway in front of all custom services.

Rationale: An API Gateway helps separate concerns: custom services can focus on solving business logic, while the gateway manages cross-cutting aspects such as authentication, API versioning, caching, rate limiting, and API documentation.

Implications:

  • Establish a clear separation of concerns by allowing custom services to focus on business logic while the API Gateway manages cross-cutting functions.
  • Enhance security through centralized authentication, authorization, and threat protection applied consistently across all APIs.
  • Promote standardization by enforcing uniform versioning, routing, and documentation for all exposed services.
  • Improve operational efficiency by leveraging built-in caching, throttling, and rate limiting to reduce load on backend services.
  • Increase observability through unified monitoring, logging, and analytics across all APIs.
  • Ownership of the API Gateway should remain with the team responsible for developing and maintaining the custom services, ensuring alignment between API design and implementation.

CHANNEL TO ENGAGEMENT

Build Backend-for-Frontend (BFF) APIs to support specific channel needs.

Rationale: Backend-for-Frontend (BFF) APIs decouple presentation from integration logic, while supporting channel-specific logic or integration protocols that don’t belong at a shared level. They also provide a secure location for storing secrets and client configurations.

Implications:

  • Promote decoupling by separating presentation-specific logic from core integration and business logic.
  • Enable channel-specific optimization so each channel — such as web, mobile, or POS — can have tailored data, logic, and protocols suited to its context.
  • Improve security by managing secrets, tokens, and client-specific configurations safely within the BFF layer.
  • Reduce complexity in frontend applications by exposing simplified, purpose-built APIs instead of direct backend integrations.
  • Increase flexibility and scalability by allowing each BFF to evolve independently without impacting other channels or consumers.
  • Acknowledge that the BFF introduces an additional component and hop in the architecture, which may affect performance; this can be mitigated by deploying it close to the frontend hosting environment and ensuring proper scaling.
  • Ensure that the BFF does not contain business logic that is generic or shared across multiple channels — such logic should reside in core services or the integration layer.

CHANNEL TO FOUNDATION

All direct integrations with Enterprise Solutions must go through API Management.

Rationale: It is generally assumed that real-time calls—and thus dependency on ERP availability and performance—are acceptable. All direct integrations with the Enterprise Solution will pass through Azure API Management. For channel-to-ERP calls, we prefer the most direct approach possible, avoiding unnecessary intermediary components.

Implications:

  • Ensure centralized security by enforcing authentication, authorization, and threat protection consistently for all enterprise APIs.
  • Promote standardization through uniform versioning, documentation, and governance across all integrations with Enterprise Solutions.
  • Provide full visibility and monitoring by enabling centralized logging, tracing, andanalytics of API traffic.
  • Support scalability and performance through caching, throttling, and load management to protect Enterprise Solutions from overload.
  • Configure Azure API Management to allow secure public connectivity while maintaining robust access controls and security configurations.
  • Enterprise Solutions must provide well-documented, stable, and versioned APIs to ensure predictable integration behavior and easier maintenance.

Integration Patterns

From the Integration Principles we can create Integration Patterns. We define Integration Patterns to help us design integrations. The Integration Patterns can be used as a 'template' for a detailed design of an integration.

Integration Patterns and Best Practices for Modern Architecture

Integration Patterns are proven, reusable solutions to common challenges that arise when connecting different systems, applications, and data sources. They describe how systems should interact, exchange data, and handle events in a consistent, reliable way.

Integration Principles vs Patterns

Integration Principles Integration Patterns
They are high-level guidelines or rules that define how integrations should be approached (e.g., “keep integrations loosely coupled,” “ensure security by design”). They are practical, reusable design templates that describe how to solve common integration problems (e.g., “Message Broker Pattern,” “Publish/Subscribe Pattern”).
Principles are why and what you should aim for — the foundation for making smart design decisions. Patterns are how you implement those principles in real-life scenarios.
Example: "Integrations should be scalable and loosely coupled." Example: "You might use the Queue-Based Load Leveling Pattern to achieve scalability and loose coupling."
Principles help you decide which patterns to use. Patterns help you apply the principles in a consistent, proven way.
